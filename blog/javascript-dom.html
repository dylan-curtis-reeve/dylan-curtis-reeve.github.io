A Non-Technical Discussion of Programming Concepts

Analogy of HTML & CSS

If you were to design your dream home HTML would be the schematics. The structure, how many rooms, & where the windows are. It would also be an itemised list of the content in the house. 3 couches, 2 beds, Washing machine, and a pizza oven. CSS on the otherhand would be the choices you make around how things look in the house. What colour are the walls, where do you want to put the 3 couches, is the washing machine going in the bathroom or the middle of the lounge?

Control Flow & Loops

You can imagine control flow and loops like the road rules. When you pull out of the car park and about to enter the road you enter a loop.

    While road isn't clear keep on checking both ways. Once road is clear turn left (or right depending on your direction).

Once you've travelled for a while you then get to a stop light. Based on the colour of the stop light you will alter your behaviour.

    If the light is red stop your car. Otherwise if the light is orange evaluate if you can stop in time or continue through if safer. In all other situations continue through the stop light.

In programming the stoplight would be like a conditional statement where you change your behaviour depending on different inputs. And there are multiple scenarios to account for.

What is the DOM and how you might interact with it

The Document Object Model (DOM) is a way for other programming languages to interact with your webpage. It provides a lot of features (or methods) for selecting parts of an HTML document and changing them based on certain criteria. The 'document' part of DOM refers to your webpage (the document). Object is a technical jargon word which signifies that this 'interface' has a bunch of properties which you can access. It's through these properties specifically that you can interact with your webpage. Model refers to the fact the document 'object' 'models' all of the components that make up a web page. This model can be visualised through a tree diagram.

Accessing Data From Arrays v Objects

The most helpful concept in distinguishing the two is that an array holds it contents in a numbered list. Now the order doesn't have to actually mean anything. But if you were to store "orange" it would be at location "x" say 3. An object on the other hand holds its content associated with a keyword. So the content of "orange" might be stored under the key "favourite_food". Which is great is you know what the keywords in an object are. But requries more work to extract content if you don't. Whereas an array is great if you don't care about what something is stored 'against' as long as it's stored. And you know what the value is which you want to retrieve.

What are Functions and Why are they Useful?

Functions are a set of instructions which have been given a name and can be accessed again and again without the programmer needing to repeat everything that goes into them each time. This is helpful as it lets you summarise 'functionality' into a name and helps to make code more readable. Not to mention save space. They can contain pretty much anything which has been discussed in this blog post to this point. And they can even contain other functions! If you've ever worked in a job with a set of operating instructions for different tasks you can think of a function as a particular task's instructions.
